- Branch: ralph/general-browser-agent (this repo was bare; init'd on first iteration).
- TS: ESM only. tsconfig has `isolatedModules: true`, so a stub file with no import/export
  must include `export {};` to be treated as a module (else top-level `const` collides
  across files in global scope).
- TS: target ES2023, module ES2022, moduleResolution `bundler`. Use `.js` extensions in
  relative imports (TS rewrites them at build, tsx accepts them at run).
- TS tests: `node --test --import tsx harness/ts/tests/*.test.ts` (no jest, no vitest).
- TS strict mode includes `noUncheckedIndexedAccess`: `arr[0]` is `T | undefined`.
  Cast at the boundary (`arr[0] as string`) or guard with `if (!x) ...`.
- CDP: do NOT pull in `chrome-remote-interface` or `puppeteer`. Node 22+ has global
  WebSocket and fetch — `harness/ts/cdp/{launch,client}.ts` is enough. Chrome is at
  `/usr/bin/google-chrome` (143). Override with `GBA_CHROME_PATH`.
- Chrome launch: `--remote-debugging-port=0` (random port), parse stderr for
  `DevTools listening on ws://...:PORT/`. Always pass `--user-data-dir=<temp>` and rm
  it on close. Headless flag is `--headless=new`.
- Process hygiene: every Chrome handle MUST expose a `close()` that SIGTERM-then-SIGKILLs
  the child and `rm -rf`s the user-data-dir. Tests assert this.
- Agent contract (US-002): `harness/ts/agent/{types,agent,trajectory,browser_session,python_bridge}.ts`.
  Every agent extends `Agent`, owns its `Trajectory`, opens it via
  `Trajectory.open({runsRoot,agent,task,seed},{agent_id,task_id,seed})`,
  and ends with `finish({terminal_state, verifier_verdict?, decline_reason?})`.
  Trajectories are JSONL `meta`/`step`/`end` lines, gzipped to `trajectory.jsonl.gz`
  on finish. The .gz is the done-marker for the resumable runner (US-010).
- Cross-language agents: TS spawns `python -m gba_agent.runner --agent <path>` via

  JSON-RPC 2.0 over stdio; `runPythonAgent(...)` registers browser/budget/trajectory
  RPC handlers and awaits a single `agent.run` call. Python proxies live in
  `harness/python/gba_agent/` and use stdlib only.
- Python: `harness/python/gba_agent/` is on PYTHONPATH via pyproject's
  `[tool.pytest.ini_options].pythonpath` for tests, and via env var set by
  `PythonAgentBridge.spawn` for the runner. No pip install needed.
- Python rpc.py: inbound RPC handlers MUST run on a worker thread so they may
  issue their own `rpc.call(...)` back to the peer without deadlocking the
  reader. Already handled inside `_dispatch_request`.
- Tests: `make test` now runs both `npm test` (TS) and `uv run pytest` (Python).

- BrowserSession.evaluate: `Runtime.evaluate` returns a result whose shape can

- BrowserPool (US-003) at `harness/ts/cdp/pool.ts`. Use `BrowserPool.create({size})`
  (default 4 / GBA_POOL_SIZE) then `pool.acquire()` / `pool.release(s)` or
  `pool.withSession(fn, {taskTimeoutMs})`. Each acquire/release destroys-and-
  respawns the slot so isolation is the contract (cookies/localStorage/cache
  are guaranteed fresh). Wall-clock deadline throws `SessionTimeoutError`
  out of session methods; agents must catch it and finish trajectories with
  `terminal_state="SESSION_TIMEOUT"` (see agents/CLAUDE.md).
- Chrome is spawned `detached:true` (own pgid) and killed via process group
  (`process.kill(-pid, sig)`) so renderer / GPU / network subprocesses die
  before `rm -rf` the user-data-dir. Without this, rm races ENOTEMPTY/EBUSY.
- snapshot/restore on PooledBrowserSession captures URL + cookies (Network.
  getAllCookies) + local/sessionStorage; restore clears, sets cookies, navigates,
  then replays storage. Pass {includeMhtml:true} for an audit DOM payload via
  Page.captureSnapshot — MHTML is NOT replayed by restore (audit only).
- Pool internals are stored in a module-level WeakMap keyed by session.
  release() does NOT delete the WeakMap entry: safeCdp() needs to read
  `deadlineHit` when the WS-close event fires AFTER release returns, so it
  can convert a "CDP session closed" rejection into SessionTimeoutError.
  Once the agent drops its session reference, GC reclaims the entry.
- LLMClient (US-004) at `harness/ts/llm/`. Construct via `defaultClient({budget,
  trajectory, paradigmSeed})` to pick up OPENAI_API_KEY / GEMINI_API_KEY from
  the env. `await llm.call(model, messages, opts)` routes by model prefix
  (gpt-*/o4-*/o3-* → OpenAI; gemini-* → Gemini). Cache lives at
  runs/.cache/llm/<key[:2]>/<key>.json keyed by sha256(stable-json({model,
  messages, opts, paradigm_seed})). mode='record' (default) calls on miss
  and persists; mode='replay' throws LLMReplayMissError on miss.
- LLM redaction is the LLMClient's job, not the provider's. Providers throw
  raw Errors that may include the API key (Gemini's URL has it as a query
  param). LLMClient catches at the boundary, replaces every `redactValues`
  occurrence with `[REDACTED]`, and rethrows. Pass `redactValues: [apiKey]`
  to LLMClient — defaultClient() does this automatically.

  cached}`. Recorded via `trajectory.recordLlmCall(...)` from inside
  LLMClient. Snapshot accessor: `trajectory.snapshotLlmCalls()`.
- Python agents reach the LLM via `gba_agent.LLMClient(rpc)` which forwards

  TS errors (LLMReplayMissError, LLMProviderUnavailableError) back to

- Verifier framework (US-005) at `harness/ts/verifier/`. Three kinds:
  `js` (CDP Runtime.evaluate in page, awaitPromise=true), `trajectory_predicate`

  `llm_judge` (temp=0, n=3 majority vote; only allowed when task tagged

  asynchronous (microtask/task), so cheats and any agent must poll
  parent's `__test` for a few frames before asserting.

- Snapshot script gotcha: <label> tags are NOT included in
  INTERACTIVE_TAGS even though they have click semantics. Including
  them shifts every input's aid by 1 (in author DOM order), which
  makes scripted-LLM tests brittle. The input itself uses
  document.querySelector('label[for="..."]') to pull the label text
  into its accessible name, so dropping label from the snapshot
  doesn't lose information.
- ReAct prompt design (US-013): one LLM turn per step, system prompt
  fully describes the JSON action schema, user message has goal +
  recent action results + current snapshot. Temperature=0 by default.
  Actions execute against `data-gba-aid` attributes (NOT CSS
  selectors), so the LLM only ever references aid integers from the
  snapshot — eliminates an entire class of hallucinated-selector
  failures.
- Empty/disabled targets are soft failures: executeAction returns
  {ok:false, message:'no element with aid=N'} which the LLM sees as
  the next observation. The agent does not abort on a single bad
  action; the loop continues for up to maxSteps. parse_error is
  similarly recorded as a step (action.type='parse_error') and
  re-prompted. Avoids dead-ends from one transient bad turn.
- Plan-then-execute agent (US-014) at agents/plan-then-execute/.

- Runtime-codegen agent (US-015) at agents/runtime-codegen/.
  Mechanism: the LLM emits raw JS function bodies that run INSIDE the
  page via Runtime.evaluate. No fixed action vocabulary. The body
  returns {done?, message, navigate?, sleep_ms?} and the agent loops.
  Distinct from baseline (fixed action set + aids) and plan-then-execute
  (fixed action set + text + batch) on the action layer itself.
- Code-emitting agents: wrap the LLM body in
  `(async () => { try { ${body} } catch (e) { return {__error:e.message}; } })()`
  so the IIFE catches runtime errors INSIDE the page and reports them
  back as structured fields. Two failure paths to remember:
   (a) syntax errors throw out of CdpBrowserSession.evaluate (parse
       happens before try/catch is entered) — agent catches at the
       evaluate() boundary, surfaces as `{error}` for next-turn
       self-correction.
   (b) runtime errors are caught by the in-page try/catch and come
       back as `{__error,__stack}`. Same surface shape after
       normaliseResult so the agent loop is uniform.
- Code-emitting agents: cross-origin top-level navigation MUST be
  requested via `{navigate: "..."}` (the agent calls browser.navigate).
  Setting `location.href = "..."` from inside the IIFE detaches the JS
  context mid-script and the in-page Promise never resolves; the
  evaluate() call hangs until the harness's per-task timeout. Document
  this in the system prompt.
- The eval runner does NOT load .env. `make eval` against agents that
  need an LLM must be preceded by `set -a && source .env && set +a`
  (or invoked from a shell with the keys already exported). When an
  LLMClient is constructed via defaultClient() and no API keys are
  in process.env, the FIRST llm.call() throws
  LLMProviderUnavailableError; agents must catch and DECLINE for the
  contract test to pass without secrets. Easy-slice cells that look
  PASS+DECLINED in the no-key path are passing because the runner
  pre-navigates to start_url and the page-text verifier matches before
  the agent does anything — this is not a regression, it's the
  baseline shape for "navigation alone suffices."
- LLM observation shape for code-emitting agents (runtime-codegen):
  structural counts (a/button/input/select/textarea/iframe/canvas/
  shadow_hosts/forms) + visible buttons (tag+text) + visible inputs
  (type+label+placeholder) + frames (src/id/name/sameOrigin) + 1500
  chars of body innerText. Crucially DIFFERENT from the baseline's
  aid-tagged element list: the LLM never references an id integer,
  because the body resolves elements with its own selectors.
- HISTORY_LIMIT for code-emitting agents should be ~4 (vs 6 for
  baseline) because each historical entry stores a ~200-char script
  excerpt PLUS the result message — more verbose per entry, so a
  shorter window stays under the prompt budget.
- Speculative-rollback agent (US-016) at agents/speculative-rollback/.
  Mechanism: every action is a SPECULATIVE TRIAL. Each step the agent
  (1) captures client-side state (URL + localStorage + sessionStorage)
  via Runtime.evaluate, (2) PROPOSER LLM emits K=2 candidate CSS-
  selector actions, (3) executes the top candidate, (4) JUDGE LLM
  (separate call) classifies the after-state as commit/revert/done,
  (5) on revert, restoreState navigates back + rewrites storage, the
  candidate's label is blacklisted, the next candidate is tried.
  Distinct from prior slots on the LOOP axis: baseline never rolls
  back, plan-then-execute commits the plan upfront, runtime-codegen
  has neither judge nor restore.
- Client-side restoreState is best-effort. Server-side mutations are
  NOT undone — if a candidate POSTed to /__submit, the submission is
  recorded server-side regardless. The judge still classifies based
  on observed state, so the mechanism keeps the agent from REPEATING
  a bad action even if it can't perfectly undo it. HttpOnly cookies
  are explicitly out of scope (document.cookie can't read them and
  we stay inside the page JS sandbox).
- Opaque origins (data: URLs) throw SecurityError on localStorage
  access. snapshot.ts wraps storage reads in try/catch so captureState
  on a data: URL returns empty maps without throwing — restoreState
  navigates back and silently no-ops on the storage write. Tests for
  storage round-trip must use a real http origin (fixtures /__health
  endpoint is fine).
- Dual-LLM pattern (proposer + judge): two SEPARATE temperature=0
  calls per attempted candidate, with different system prompts. The
  judge sees {goal, state_before_digest, action_label, executor_result,
  state_after_digest} and returns {verdict, reason}. Worst-case cost
  per step with K=2 and one revert is 3 LLM calls (1 propose + 2
  judge); on success it's 2.
- Per-state blacklist: when a candidate's judge verdict is `revert`,
  the canonical action label (e.g. `click(button#submit)`) goes into
  a blacklist surfaced to the next proposer turn. On `commit` the
  blacklist resets (we're in a new state). Capped at BLACKLIST_LIMIT
  entries to keep prompts bounded.
- Agent IDs MUST be unique in agents/ for auto-discovery; the directory
  name IS the id and the manifest.id must match. data-* attributes
  used by agent snapshots SHOULD also be unique (data-gba-aid is taken
  by baseline, data-rollback-aid was avoided — speculative-rollback
  does NOT tag elements in the DOM at all; it relies on CSS selectors
  from LLM-authored output, which is itself a distinguishing choice).
- Predicate-driven agent (US-017) at agents/predicate-driven/.
  Mechanism: ONE upfront LLM call synthesises a JS predicate that
  returns true in the page when the goal is met; the loop terminates
  from CODE the moment the predicate evaluates true. The action LLM
  has NO `finish` action — actions.parseAction explicitly rejects it
  with a message about the agent's invariant. Distinct on the
  TERMINATION axis from all four prior agents (baseline finish,
  plan-then-execute end-of-plan, runtime-codegen body.done, srb judge).
- predicate.ts: parsePredicate accepts {predicate|expression|expr|test}
  + optional {rationale|thought|why}; wrapPredicate produces
  `(async () => { try { return Boolean(<expr>); } catch (e) {
   return { __predicate_error: ... }; } })()` so syntax errors fall
  out of evaluate() (parse fails before try) and runtime errors come
  back as a typed object. evaluatePredicate normalises both paths into
  {satisfied: bool, error?: string}. Wrapping uses Boolean() so any
  truthy/falsy value collapses cleanly.
- Two-LLM-call agents that share an LLMClient + paradigmSeed CAN
  produce identical cache keys if both calls have identical messages.
  Predicate-driven dodges this by giving the synthesiser a different
  system prompt (PREDICATE_PROMPT vs ACTION_PROMPT), so the two calls
  always hash to different keys even if the user content matched.
- Test pattern for a "loop terminates from code" agent: don't end
  scripted-LLM turns at the boundary you expect to hit. The
  scriptedProvider stays parked on the LAST turn so an unexpectedly-
  short loop will burn LLM calls but not throw "out of turns" — see
  the "predicate stays false through maxSteps" test.
- LLMMessage.content (US-018) is now `string | LLMContentPart[]` so
  multimodal calls (OpenAI vision: text part + image_url part) round
  trip through the harness. `LLMContentPart` is exported. The Gemini
  provider explicitly throws on array content — only OpenAI is wired
  for vision. Cache key (sha256 over stable-sort JSON) handles arrays
  naturally; identical multimodal payloads round-trip record→replay
  with cached=true (vision_grounded test asserts this).
- OpenAI provider (US-018) gained transient-error retry: 429 + 5xx
  trigger up to 5 retries with backoff = max(Retry-After header,
  body's "try again in Xms" hint, exponential 500ms × 2^attempt
  capped at 30s). Non-transient 4xx (e.g. 401) are NOT retried.
  Vision agents need this — TPM limits get hit fast when each call
  ships a screenshot. Tests at llm_providers.test.ts cover the
  transient/non-transient split + max-attempts give-up.
- Vision-agent gotcha: when the visible page state doesn't change
  between actions (e.g. a click that fires a fetch but the response
  isn't painted yet), JPEG bytes are bit-identical → cache key
  matches → LLM returns cached "do the same thing again" → infinite
  loop. Fix: include the step counter in the user prompt
  (vision-grounded does this in buildMessages) so each call has a
  fresh cache key. Also add a small (200ms) post-action settle
  before re-observing so async fetch responses + paint actually land.
- For an OpenAI vision call, image_url.detail trades token cost vs
  localisation accuracy: "low" is a flat 85 tokens (good for TPM,
  bad for small targets); "high" is 85 + (n_tiles × 170) on a 512×512
  tile grid (~765 tokens for 800×600, much better resolution). For
  vision agents that need to click anything smaller than ~150px,
  "high" is required. "auto" can pick either based on size.
- Pixel-localisation accuracy of generic vision LLMs (gpt-4o-mini,
  gpt-4o) on novel page layouts is poor — both models center-bias
  their x-coordinate estimates. Empirical evidence in
  runs/vision-grounded/hard-recoverable/0/trajectory.jsonl.gz:
  agent's thought is "click Submit order"; emitted coords are
  (390,220); actual button is at ~(110,170). This is a known
  research finding (WebVoyager's Set-of-Marks paper, ScreenSpot
  benchmark). Do NOT expect a generic chat model to do un-augmented
  pixel grounding well. Future vision agents should consider:
  Set-of-Marks overlays (defeats the no-DOM-augmentation claim),
  coordinate-grid overlays (preserves it but reframes the agent),
  or purpose-trained GUI models (CogAgent, OS-Atlas) plumbed in via
  a new LLMClient provider.
- Network-shadow agent (US-019) at agents/network-shadow/. Mechanism:
  installs a fetch + XMLHttpRequest monkey-patch via BOTH
  Page.addScriptToEvaluateOnNewDocument (covers future docs/popups) and
  Runtime.evaluate (covers the already-loaded doc since the harness
  navigates to start_url before agent.run()). Every request the page
  or the agent makes lands on window.__gba_net_log capped at 60 entries.
  LLM action vocabulary is HTTP-first: `fetch(method, url, body?,
  content_type?)` executed IN-PAGE so cookies+origin are preserved, plus
  click/navigate/wait/done/decline as fallbacks. Distinct from every
  prior slot on TWO axes (observation = network traffic; action = HTTP
  requests). Live results: 21/22 easy, 3/10 hard (shadow-form,
  recoverable, modal-stack).
- Monkey-patch idempotency: every install path MUST guard with a
  window-level `__installed` flag before re-wrapping
  fetch/XMLHttpRequest, or a second install will wrap the wrapped
  function and every request gets logged twice. installPatch() also
  has to tolerate failures from Page.addScriptToEvaluateOnNewDocument
  (the Page domain may not be enabled depending on session lifecycle)
  AND Runtime.evaluate (opaque origins / about:blank). Both calls are
  wrapped in try/catch with silent swallow — the patch is best-effort
  and reinstalls automatically on every navigate action.
- XHR monkey-patch detail: when wrapping XMLHttpRequest, copy the
  prototype AND the constructor's own properties via
  Object.setPrototypeOf(Patched, Orig). Without the prototype copy
  pages that pass an XHR instance to `instanceof XMLHttpRequest`
  check (e.g. libraries that detect "is this a real XHR") will
  return false and bail. Tested by leaving the existing fixtures-
  page fetch logic intact and asserting all requests still complete.
- "Trigger UI to discover the endpoint" pattern: when the LLM doesn't
  yet know the endpoint behind a form, it can `click(button#submit)`
  to make the PAGE issue its own fetch, see it in the log, and then
  on the next turn replay or manipulate that request. This is the
  network-shadow agent's recovery path for pages whose endpoints are
  not visible in form action="" attributes (e.g. JS-only submitters,
  shadow-rooted forms). The agent picked this strategy in trajectory
  runs/network-shadow/hard-shadow-form/0/ — first click bounced because
  the shadow root wasn't reachable, but the page's wrapping JS made
  the right call which then showed up in the log; the agent replayed
  the POST directly on the next turn and won.
- Action substrate "HTTP requests" matters distinctly from "CSS
  selectors". The win on hard-shadow-form proves the design: the
  shadow root is irrelevant once the agent can see the endpoint, and
  the endpoint is same-origin so cookies+CSRF (if any) come along

    process group (`process.kill(-pid, signal)`). Without this, the
    renderer/GPU/network subprocesses keep writing into --user-data-dir
    after the parent dies and `rm -rf` races with ENOTEMPTY / EBUSY. This
    fixed both the new pool test AND a pre-existing flake in
    smoke.test.ts ("temp user-data-dir is removed on close").
  - When the parent chrome is already dead (e.g. external SIGKILL in the
    crash-replace test), still call killGroup(SIGKILL) before rm — group
    members survive the leader. killChild() now does this.
  - Acquire/release-then-respawn is the only reliable way to guarantee FR-4
    isolation without a per-slot scrub script that future agents would
    forget. ~1s spawn cost is acceptable for the research harness.
  - `Network.enable` MUST be sent at slot init for snapshot/restore (cookie
    domain methods are gated on it). Page.enable alone is not enough.
  - Cookies don't persist on data: URLs (opaque origin), so any pool
    snapshot/restore test needs an http origin. The test file spins up a
    tiny http.createServer per test on 127.0.0.1:0 and tears it down at
    the end. Pattern: `startFixtureServer()` returning {origin,url(),close()}.
  - `assert.rejects(() => stat(dir), /ENOENT/)` is the right assertion for
    "this directory was removed". stat() throws an Error whose .message
    starts with "ENOENT: no such file or directory, stat '...'" — the
    regex matches.
  - rm({maxRetries:5, retryDelay:100}) is a useful safety net even with
    pgid-killing — keep it. Linear backoff on EBUSY/ENOTEMPTY/EMFILE/etc.
  - Pool wakeWaiter() runs in a tight while-loop pairing waiters with
    idle slots; safe because both arrays mutate synchronously inside the
    loop. Don't add awaits inside that loop or you'll get double-wakes.
  - PooledBrowserSession exposes `userDataDir` and `pid` getters (read
    through the WeakMap). Useful for tests; agents shouldn't need them
    in normal operation but they don't leak anything sensitive.
  - The hang-agent test had to await its inner trajectory-write Promise
    AFTER the outer `pool.withSession` race rejected — otherwise the .gz
    file might still be in flight when the test asserts on it. Pattern
    captured in `runHangAgent` helper inside pool.test.ts.

- **Learnings for future iterations:**
  - PDFs do NOT need a heavy library. A 5-object PDF (catalog,
    pages, page, contents, font) with a single Tj text run is ~400
    bytes and works in real viewers. The hard parts are: byte
    offsets must be byte-counted not char-counted (use
    Buffer.byteLength), the binary marker on line 2 (%\xFF\xFE\xFD\xFC)
    needs `Buffer.from(s, 'binary')` so Latin-1 bytes survive, and
    the xref table padding is exactly 10 digits. The code lives
    in pages/pdf_task.ts as buildAnswerPdf(answer) — reusable for
    any future fixture that needs a tiny PDF.

    gesture in headless Chrome (DOM activation). For a real agent
    that needs window.open() to succeed, the click must come from
    JS — but our cheats avoid that path entirely by using the
    server-side token endpoints. CdpBrowserSession only attaches
    to the first target, so popups run their own JS context that
    we cannot evaluate into. The clean pattern is "popup
    auto-postback + opener.postMessage": no CDP target plumbing
    needed for the parent flow.
  - For per-session state that needs to rotate on /__reset (random
    answers, retry counters, token-scoped codes), put the
    rotation INSIDE freshState() so Object.assign(state,
    freshState()) handles it uniformly. We deliberately use a new
    Map() reference rather than calling .clear() on the old one;
    no callers retain a reference to the old Map past
    Object.assign.
  - When a fixture has BOTH a "race / fail-then-retry" path and a
    success path, encode the failure mode as a status flag on the
    page (button.disabled, error-banner textContent) so cheats and
    real agents can poll on it — don't rely on timing or fetch
    return values directly. Recoverable's cheat polls attempts===1
    AND !btn.disabled before retrying, which is robust against
    fetch-resolution timing.
  - The trivial agent's DONE-but-FAIL outcome on pdf-task (it
    navigates to the only `<a href>` on the page, /report.pdf, and
    declares DONE) is a useful regression test: terminal_state and
    verifier verdict are independent. Tournament metrics MUST use
    the verifier verdict, not the agent's self-report. A future
    agent that prematurely declares DONE on this fixture should
    score 0.
  - Late-hydration's removeEventListener-then-addEventListener
    handler swap is the cleanest way to model "the JS handler
    arrived later" without fragile race conditions in the no-op
    handler. Don't try to put the gating logic INSIDE the handler
    (e.g. `if (!hydrated) return;`) — that lets agents accidentally
    succeed on the first post-hydration click that is part of a
    burst, which weakens the AC. Removing the handler entirely
    means pre-hydration clicks are observably no-ops.

- **Learnings for future iterations:**
  - When designing an agent whose mechanism is "loop terminates on a
    code-evaluated condition", the EASIEST way to keep the LLM from
    short-circuiting is to refuse the `finish` action at the parser
    level, not in the prompt. parseAction in actions.ts throws
    ActionParseError with a message that explicitly cites the
    agent's invariant; the loop records a parse_error step and
    continues. Prompts can drift; parsers cannot.
  - Wrapping an LLM-emitted JS expression for in-page eval has TWO
    failure paths that need separate handling:
     (a) syntax errors throw OUT of CdpBrowserSession.evaluate
         because parsing happens BEFORE the try/catch wrapper runs;
         catch at the evaluate() boundary in TS.
     (b) runtime errors are caught by the in-page try/catch and
         come back as a structured object ({__predicate_error}).
    evaluatePredicate normalises both into the same
    {satisfied, error?} surface — that single shape simplifies the
    agent loop a lot.

    should warn against being true at start. An eager predicate
    that fires before any action makes the run trivially DONE with
    no work — it's not technically wrong (the verifier may even
    match) but it makes the trajectory uninformative for debugging.
  - The agent's predicate is INDEPENDENT of the harness verifier
    (harness/ts/verifier/). They may even disagree: the agent may
    DECLINE on a max-steps limit while the page state actually
    matches the verifier. This is fine — the agent's predicate is
    a search heuristic, the verifier is the scoring oracle.
  - Two LLM calls in the same agent that share an LLMClient +
    paradigmSeed CAN produce identical cache keys if both calls have
    identical messages. Predicate-driven dodges this by giving the
    synthesiser a different system prompt (PREDICATE_PROMPT vs
    ACTION_PROMPT), so the two calls always hash to different keys.
  - When testing a "loop terminates from code" agent, leave the
    scriptedProvider parked on its LAST turn rather than throwing
    out-of-turns. An unexpectedly-short loop should burn LLM calls
    silently rather than crash the test — see the
    "predicate stays false through maxSteps" test.

    code path's type narrowed inside the function.

    cache.ts's stableSort recurses into nested objects. No special-
    casing needed for content arrays. Test this explicitly with a
    record-then-replay-with-same-payload assertion (see
    vision_grounded_agent.test.ts "LLMClient: cache key handles
    multimodal content").

    the agent gets stuck in an infinite loop. Two fixes that

    step counter in the user prompt so cache keys differ across
    steps even when the screenshot is unchanged, (b) add a

- **Learnings for future iterations:**

    cover EVERY future document the renderer creates (page
    navigations + popups). To also cover the CURRENT document
    (which the harness has already navigated before agent.run()),
    a parallel Runtime.evaluate install is required. Both calls
    MUST tolerate failure (the Page domain may not be enabled,
    the current document may be opaque-origin) — wrap in try/catch
    and silently swallow. The patch will reinstall itself on every
    navigate action so a single failure isn't fatal.

    `window.__patch_installed = true` flag at the top of the IIFE
    and short-circuit if already true. A second install on a
    wrapped fetch produces a wrapper-of-a-wrapper that logs every
    request twice.

    PatchedXhr.prototype = OrigXhr.prototype + Object.setPrototypeOf
    (Patched, Orig). Pages that do `xhr instanceof XMLHttpRequest`
    will fail without the prototype copy. The fixtures-server
    fetch path doesn't exercise this but third-party libraries
    might.

    agents: cookies + same-origin auth + CSRF tokens (if any) ride
    along automatically because the request runs in the page's
    own JS context. Trying to issue requests from the Node side
    (via fetch in the agent process) would lose all of that AND
    require manually copying document.cookie.
  - When the LLM doesn't know an endpoint, the "trigger UI to
    discover" pattern works: click(button#submit) makes the PAGE
    issue its own fetch, which lands in the network log; the next
    LLM turn can replay or manipulate that request. This is the
    agent's recovery path for forms whose action="" isn't set
    (the JS-only submitters that dominate modern SPAs).
  - The fixtures server's /api/buy + /api/last pattern is a nice
    test harness for network-aware agents: a stateful endpoint
    (records last submission) lets a unit test assert both the
    agent's response handling AND the server-side side-effect
    after the run.
  - For the e2e tests, the `installPatch is idempotent` check is
    a great regression guard. Double-installing on a real page is
    the easiest way to trip silently into "every request shows up
    twice in the log", which then poisons the LLM's prompts.

  every childList/attribute/characterData mutation onto
  window.__gba_dom_log with a strictly-monotonic seq. Each step the
  LLM sees the mutation tail (most recent 24) alongside a minimal
  aid-keyed interactive-element snapshot, then emits one of
  {click(aid), type(aid, text, submit?), scroll(direction, pixels?),
  wait(ms), await_change(timeout_ms), navigate(url), done, decline}.
- After every state-changing action the harness calls settleAfter()
  which BLOCKS in-page until the mutation log grows AND quiesces for
  ≥75ms (cap 400ms) — the LLM only ever observes a settled
  post-action state, not a mid-reaction view. The await_change
  action is a first-class "block until the page moves" primitive
  that no prior agent has — turn cadence is gated by the document,
  not wall-clock.
- Distinct on TWO axes from every prior slot:
   * Observation modality: DOM mutation deltas (this) vs DOM/a11y
     snapshots (baseline, plan-then-execute, runtime-codegen,
     speculative-rollback, predicate-driven) vs pixels
     (vision-grounded) vs network traffic (network-shadow).
   * Cadence primitive: await_change blocks until a mutation arrives
     (or quiesces after a state change); prior agents are all
     wait(ms) wall-clock-only.
  approach_keywords = [mutation_observer, dom_event_stream,
  delta_observation, change_driven_loop, await_dom_change,
  transition_aware]; Jaccard=0 with every prior agent (explicit
  test in the test file).
- Composition: reuses install pattern from network-shadow (add-on-
  new-doc + current-doc eval + idempotency guard) and aid-keyed
  action substrate from baseline (data-gba-stream-aid, distinct
  attribute name so multiple agents in the same harness run don't
  collide). The COMPOSITION is novel because the observation axis
  (deltas, not states) and the cadence primitive (await_change)

  new perception+action shape.

- **Learnings for future iterations:**
  - MutationObserver entries fire on a microtask, not synchronously
    with the DOM operation that produced them. The settleAfter()
    pattern (poll seq for change, then wait for quietMs of no
    change) is the right way to know "page has finished reacting"
    — a single wait(ms) is too brittle, and reading the log
    immediately after a click misses the mutation that the click
    just triggered.
  - The CHARACTER of the mutation stream changes by fixture type:
    modal-stack emits one or two `+ div` / `- div` per step
    (clean FSM signal); recoverable emits attr flips on the
    button's `disabled` (clean retry signal); virtual-scroll
    emits *hundreds* of row +/- entries (noise). Capping the log
    at 200 with FIFO drop is the right policy because the AGENT
    is reading the tail anyway, but a future iteration could
    add a coalesce pass (collapse N consecutive `+ tr` into
    `+ tr ×N`) to compress the noisy case.
  - Use a DISTINCT data-attribute name (data-gba-stream-aid here,
    vs baseline's data-gba-aid) when an agent re-implements a
    DOM-tagging convention. Sharing the attribute would mean
    multi-agent runs collide, and tagging in different orders
    would shift aids across agents.
  - The `await_change` action's defaults are load-bearing: if the
    LLM emits `{"type":"await_change","timeout_ms":0}` we promote
    it to 1500ms rather than no-op. The LLM's intent is "I expect
    a slow change, don't observe yet" — a 0ms await_change has
    no good interpretation, so the parser DTRT.
  - When recording action results into the trajectory, including
    `mutation_delta` per step turned out to be the cleanest
    debugging surface for "is the agent actually seeing the
    change?" — modal-stack's Δmut=7,1,5,0 trace immediately
    shows which step advanced the FSM and which clicked an
    already-active button.
  - The harness has no .env auto-loader — `make eval` runs under
    process.env directly. To run a live eval, source .env first:
    `set -a && . .env && set +a && make eval AGENT=... SLICE=...`.
    Without it the agent will land in the "no LLM provider
    configured" decline path on every task in under 1s. Worth
    adding `--include` of dotenv to the eval CLI in a future
    iteration.

- **Learnings for future iterations:**
  - Docker volume permissions on Linux: a freshly-created named
    volume is owned by root. Containers that run as a non-root
    user (vikunja's uid 1000) need either a chown init container
    or `user: "0:0"` override. The init-container pattern keeps the
    main service runtime clean; document it in compose comments so
    future refactors don't accidentally remove it.
  - Always check the actual image's expected env var names against
    the docker-compose.yml — image vendors disagree on shorthands.
    bookstack expects `DB_USERNAME`, mariadb expects `MARIADB_USER`,
    gitea expects `GITEA__database__USER` (double underscore).
    A wrong env var name silently falls back to defaults, which
    then 500's the app on first request.
  - APP_URL / hostname matching matters even for local dev because
    web apps use it for 302 Location headers and cookie domains.
    Mismatching the navigate host and APP_URL turns same-origin
    fetches into cross-origin, which fails with a confusing
    "TypeError: Failed to fetch" message.
  - The verifier wrap bug went undetected because every existing
    task spec used a SINGLE-LINE expression (no trailing `;`).
    When introducing a new task author convention (multi-line
    block scalars), validate it through the full pipeline before
    shipping a slice that relies on it. A unit test that wraps a
    representative expression shape would catch this in seconds.
  - State contamination in stateful slices is a real concern.
    Either reset state between cells/agents OR design verifiers
    that check for per-agent provenance. The "trivial agent gets
    4/9" smoking gun in this sweep was an unmistakable signal.
  - The tournament runner's resumable per-cell pattern works well
    in practice: I ran 4 parallel sweeps on the same runs-root,
    each with --agents filtered to one agent, and the per-agent
    subdirectories prevented any conflicts. A subsequent unified
    `make tournament SLICE=hard-app` discovered all agents AND
    correctly resumed every existing cell (no re-runs). This is
    a battle-tested invariant now.
  - When sweeping many agents in parallel, watch the OpenAI TPM
    rate limit. The predicate-driven sweep hit a 429 on the last
    cell because 4 sweeps were all blasting gpt-4o-mini calls
    simultaneously. The harness handles 429 as `ERROR` terminal
    state, not a crash, which is the right behaviour — but cell-
    level data is lost. The 429-with-backoff in
    harness/ts/llm/providers/openai.ts (added in US-018) helped
    most cells absorb transient limits.
  - vision-grounded's 0/10 → 5/9 transfer was the most surprising
    result of the sweep. The published WebVoyager/SeeAct/AppAgent
    centre-bias failure mode is most pronounced on tightly-packed
    UIs with small target buttons; real CRUD apps (gitea forms,
    vikunja tasks) have generous-sized buttons in predictable
    positions, and gpt-4o-mini's pixel-coord clicks land reliably
    enough to win 5/9. This is a strong argument for the hard-app
    slice as the better generalisation test going forward, AND
    a counter-argument to the US-031 "vision agents are dead
    without SoM" framing (the verdict depends heavily on what
    the UI looks like).

- **Learnings for future iterations:**
  - The `pattern:<id>` tag pattern is a strong "no duplicate fixture"
    enforcer that's also greppable. Future slices that want similar
    breadth guarantees should adopt this convention rather than
    relying on (host, skill) uniqueness — multiple navigate tasks
    on the same host are legit if their mechanism (text-search vs
    tab vs inline-link) differs.
  - For real-site canaries that risk passing click-first-link via
    skip-to-content anchors, add `document.location.hash === ''` to
    the verifier. Most LLM-using agents don't add a hash mid-run, so
    this is safe. It is the simplest knob to push the trivial agent's
    pass rate down without making canaries materially harder.
  - The pre-US-029 slice had a structural signal-compression bug:
    the eval runner pre-navigates to start_url BEFORE agent.run(),
    so any agent that declines or no-ops sees its verifier run on
    the start_url's body. Body-text-only verifiers therefore pass
    universally on the easy slice — the bar is "did the test rig
    do its job", not "did the agent do anything". The fix is to
    require URL-shape change in the verifier; this is the same
    pattern hard-real (US-026) already uses.
  - Real-site latency varies a lot. Wikipedia is ~1s/page,
    arxiv.org is ~2-3s, github.com is ~3-5s, MDN is ~2s, info.cern.ch
    is ~50ms. Easy budget (60s wall) is enough for 5-15 sequential
    page loads but NOT for vision agents that ship a screenshot per
    step on a 4-step task — vision-grounded routinely consumed 11s
    mean cell latency on the v2 slice and still timed out on dense
    UIs because the LLM call alone is 5-8s.
  - Parallel tournament processes on one shared runs-root work
    cleanly: per-agent subdirectories prevent contention, and the
    per-cell summary.json done-marker is atomic. 8-way parallelism
    on this workstation kept all cores warm without hitting OpenAI
    TPM limits at the model's default rate (gpt-4o-mini). Pattern
    is the same as the US-027 iter-2 hard-app sweep.
  - When picking the click-first-link FAIL invariant, the safest
    canary verifier shape is:
       `pathname === '<exact_path>' && hash === '' && /<body_text>/i`
    The exact-pathname plus empty-hash combination kills both
    skip-anchor hash adds and full-URL changes. Body-text alone
    is not enough; URL-shape alone is not enough either (because
    some bodies are dynamically loaded by JS after navigate).
