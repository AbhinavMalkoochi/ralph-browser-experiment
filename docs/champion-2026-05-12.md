## Mechanism

Source: [`agents/runtime-codegen/`](../agents/runtime-codegen/README.md),
manifest: [`agents/runtime-codegen/manifest.yaml`](../agents/runtime-codegen/manifest.yaml).

Each turn the LLM emits the **body of an `async` JavaScript function** that
runs inside the page via CDP `Runtime.evaluate`. There is no fixed action
vocabulary; the body has first-class access to the DOM (shadow-root
traversal, same-origin iframe `contentDocument`, synthetic `MouseEvent`
dispatch on `<canvas>`, `fetch()` to same-origin endpoints, `postMessage`,
…). The body returns `{done?, message, navigate?, sleep_ms?}` and the
harness loops until `done` or step cap.

The pertinent distinguishers from prior slots:

* `baseline-a11y-react`: fixed JSON action set keyed by `data-gba-aid`. Cannot
  reach shadow roots or canvases without a new action type.
* `plan-then-execute`: text-keyed batch plan, no in-page code.
* `predicate-driven`: action LLM has no `finish`; termination is code, but
  actions still come from a fixed vocabulary.
* `network-shadow`: HTTP-first vocabulary. Wins on shadow-form by replaying
  the page's own POST, but cannot draft synthetic canvas events.

The shared failure axis is **multi-context coordination**: anything that
requires reasoning across iframes, popups, or non-HTML responses defeats a
single in-page IIFE. This argues for the US-024 meta-mixture experiment to
route those fixtures to `network-shadow` (PDFs reachable as bytes) and a
future popup-aware agent rather than asking `runtime-codegen` to solve them.
